# 5etools Personal Development Guide

> Generated for solo developer use. All recommendations reference actual files and patterns observed in this repo (v2.25.3).

## SECTION 1 — Codebase Architecture Overview

### Top-Level Folder Structure

| Directory | Purpose |
|-----------|---------|
| `js/` | All application JavaScript (164 files). Page controllers, renderers, filters, utilities. |
| `data/` | D&D content as JSON files. Organized by entity type (bestiary, class, spells, etc.). |
| `css/` | Compiled CSS output (34 files). **Do not edit directly** — generated from `scss/`. |
| `scss/` | SCSS source files (28 main + 62 partials in `includes/`). Compile to `css/`. |
| `node/` | Build scripts and code generators (42 files). Node.js tooling for search indexes, pages, data processing. |
| `lib/` | Vendored third-party libraries (ace.js, elasticlunr.js, localforage.js, peerjs.js, etc.). |
| `test/` | Test suites — JSON validation, tag checking, unit tests (Jest). |
| `search/` | Pre-built search indexes (JSON). Generated by `node/generate-search-index.js`. |
| `homebrew/` | Local homebrew content system. Contains `index.json` pointing to user-added JSON files. |
| `fonts/` | Web fonts for the UI. |
| `icon/` | SVG/PNG icons. |
| `audio/` | Audio assets. |
| `bestiary/`, `items/`, `spells/` | Sub-page index.html redirects for deep-linking. |
| `prerelease/` | Pre-release content staging area (same pattern as `homebrew/`). |

### Primary Entry Points

- **`index.html`** — Main landing page. Loads navigation, search, and the page index.
- **`spells.html`, `bestiary.html`, `items.html`, `classes.html`, etc.** — Each HTML page is a standalone SPA-like view for one entity type.
- **`5etools.html`** — Alias/redirect entry point.
- **`dmscreen.html`** — DM Screen tool (complex multi-panel app).

### Architectural Pattern

**Multi-page application with shared core libraries.** Each HTML page loads a common set of scripts via `<script>` tags in a specific order:

1. **Bootstrap layer:** `styleswitch.js`, `navigation.js`, `browsercheck.js`
2. **Core utilities:** `parser.js`, `utils.js`, `utils-ui.js`
3. **Storage:** `localforage.js` (IndexedDB wrapper for offline data)
4. **Search/filter:** `omnidexer.js`, `omnisearch.js`, `filter.js`
5. **Data loading:** `utils-dataloader.js`, `utils-list.js`
6. **Rendering:** `render.js`, `render-dice.js`, `render-markdown.js`
7. **Page framework:** `listpage.js` (shared list/detail page logic)
8. **Page-specific:** e.g., `filter-spells.js`, `spells.js`, `render-spells.js`

This is **not** a framework-based SPA. There is no React, Vue, or Angular. It is a **vanilla JS class-based architecture** with ES modules (`"type": "module"` in package.json). Classes use inheritance (e.g., `SpellsSublistManager extends SublistManager`).

### Library Dependencies

- **No runtime npm dependencies** — all dependencies are devDependencies for build tooling.
- **Vendored client-side libs** (in `lib/`): localforage, elasticlunr, ace editor, peerjs, lzma, polylabel.
- **Build tools:** sass, esbuild, eslint, stylelint, prettier, jest, workbox-cli, ajv (JSON schema validation).
- **Node.js utilities:** 5etools-utils, commander, handlebars, simple-git, sharp (image processing).

### Organically Grown / Inconsistent Areas

- **`js/parser.js`** (4,508+ lines): Monolithic parser with many static methods. Acts as a god-object for string parsing, entity resolution, and data normalization.
- **`js/utils.js`** (9,551 lines): Extends `String.prototype` globally (uppercaseFirst, toTitleCase, etc.). Mixes global constants (`VeCt`), version tracking, and utility functions in one file.
- **Filter files vary wildly in size:** `filter-bestiary.js` (29,903 bytes) vs `filter-common.js` (4,525 bytes). The filter system is powerful but dense.
- **Line counts in comments are byte counts**, not actual line counts for some files — the build system bundles/minifies some JS files.

## SECTION 2 — Build System & Local Dev Workflow

### Build System

**SCSS → CSS:** Sass compiler (`sass --style=compressed scss/:css/`)
**JS:** No bundler for client-side JS. Files are loaded directly via `<script>` tags. `esbuild` is used only for the service worker build.
**Data generation:** Custom Node.js scripts in `node/` generate search indexes, lookup tables, and derived data.
**Service worker:** Built with `workbox-cli` via `node/build-sw.mjs`.

### Commands

```bash
# Install dependencies
npm install

# Run locally (serves on port 5050)
npm run serve:dev

# Build everything (clean JSON, generate data, compile CSS, build service worker)
npm run build

# Individual build steps
npm run build:css          # Compile SCSS to CSS
npm run gen                # Run all data generators
npm run gen:search-index   # Rebuild search indexes only
npm run gen:pages          # Regenerate HTML pages

# Testing
npm test                   # Full test suite (lint + unit + CSS + data)
npm run test:js            # ESLint only
npm run test:unit          # Jest unit tests
npm run test:data          # Data validation

# Linting/formatting
npm run lint               # Fix all lint issues
npm run lint:js            # ESLint autofix
npm run lint:css           # Stylelint + Prettier autofix
npm run lint:data          # Prettify JSON data files
```

### Watch / Hot-Reload

**There is no built-in watch mode.** The dev server (`http-server`) serves static files with no caching (`-c-1`) and CORS enabled. To see changes:

- **JS changes:** Refresh the browser (no compilation needed — raw JS served directly).
- **SCSS changes:** Run `npm run build:css` then refresh.
- **Data changes:** May require `npm run gen` if generated data is affected, otherwise just refresh.

**Tip:** For SCSS watch, run sass with `--watch` manually:
```bash
npx sass --watch scss/:css/
```

### Environment Variables / Config

- **`.node-version`**: Specifies Node.js 17.1.0. Use nvm/fnm to match.
- **No `.env` file** required. The app runs as static files.
- **`IS_DEPLOYED`** flag in `js/utils.js`: Controls homebrew loading. Set to `undefined` to enable local homebrew features.

### Known Gotchas

1. **Node 17 requirement:** The `.node-version` says 17.1.0. Newer Node versions may work but aren't guaranteed. If you see crypto or module errors, check your Node version.
2. **No img/ directory in repo:** Images are gitignored and served from an external source or a separate image repo (see the Docker image `5etools-img`).
3. **Generated files must exist:** The `data/generated/` directory contains build artifacts. Run `npm run gen` after cloning before the site will work fully.
4. **Search indexes are large:** `search/index.json` is ~2MB, `search/index-foundry.json` is ~3MB. Regenerating them takes time.
5. **The `css/` directory is build output:** Don't edit CSS files directly. Edit the corresponding `.scss` file in `scss/` or `scss/includes/`.

## SECTION 3 — Data Architecture

### Data Structure

All game content lives in `data/` as flat JSON files organized by entity type:

- **Single-file entities:** `actions.json`, `backgrounds.json`, `feats.json`, `items.json`, `races.json`, `languages.json`, `optionalfeatures.json`, etc.
- **Multi-file entities** (split by source book):
  - `data/bestiary/bestiary-phb.json`, `bestiary-xge.json`, etc. (204 files)
  - `data/class/class-wizard.json`, `class-cleric.json`, etc. (35 files)
  - `data/spells/spells-xphb.json`, etc.
  - `data/adventure/` and `data/book/` — adventure/book content split by source
- **Fluff files** (flavor text, images, lore): `fluff-items.json`, `fluff-races.json`, etc.
- **Foundry VTT exports:** `foundry-items.json`, `foundry-feats.json`, etc.
- **Generated data** (in `data/generated/`): Lookup tables, cross-reference indexes, DM screen references. Built by `npm run gen`.

### JSON Schema Pattern

Each data file follows a consistent top-level structure:
```json
{
  "spell": [
    {
      "name": "Fireball",
      "source": "PHB",
      "page": 241,
      "level": 3,
      ...
    }
  ]
}
```

The top-level key matches the entity type (plural or singular depending on the entity). Every entity has `name` and `source` fields. The `source` field is an abbreviation (PHB, XGE, TCE, etc.) mapped in `parser.js`.

### Homebrew / Custom Content System

**Location:** `homebrew/index.json`

The homebrew system allows adding custom JSON content files:

1. Create a JSON file in `homebrew/` following the same schema as the official data files.
2. Add the filename to the `"toImport"` array in `homebrew/index.json`.
3. Ensure `IS_DEPLOYED` is set to `undefined` in `js/utils.js` (it's disabled in production builds).

**The prerelease system** (`prerelease/`) works identically but for pre-release/playtest content.

The homebrew manager UI is at `managebrew.html` and the homebrew creator at `makebrew.html`.

### Data → UI Rendering Pipeline

1. **HTML page loads** → page-specific JS (e.g., `spells.js`) initializes
2. **DataLoader** (`utils-dataloader.js`) fetches JSON from `data/`, merging official + homebrew + prerelease content
3. **Parser** (`parser.js`) normalizes entities (e.g., CR values, spell levels, source lookups)
4. **Filters** (`filter-spells.js` etc.) build filter UI from data properties
5. **List** (`list2.js`, `listpage.js`) renders the left-panel entity list with search
6. **Renderer** (`render.js` + `render-spells.js` etc.) renders selected entity detail into the right panel `#pagecontent`

### Best Entry Points for Adding/Modifying Content

1. **Adding homebrew content:** Create JSON files in `homebrew/` matching the official schema. No code changes needed.
2. **Adding a new data source:** Add a new JSON file to the appropriate `data/` subdirectory and update the relevant index.
3. **Modifying display of existing entities:** Edit the corresponding `render-*.js` file (e.g., `render-spells.js` for spell display).
4. **Adding new filter options:** Edit the corresponding `filter-*.js` file.

## SECTION 4 — Safe Modification Zones

### Safe to Modify (Low Risk)

| File/Directory | Why Safe |
|---------------|----------|
| `homebrew/*.json` | Designed for user content. Completely isolated from core. |
| `homebrew/index.json` | Just a pointer file — add your filenames here. |
| `scss/includes/*.scss` | Scoped partials. Changes affect only their specific component. |
| `scss/{page}.scss` | Page-specific styles (e.g., `scss/bestiary.scss`). Isolated scope. |
| Individual `render-*.js` files | Each handles one entity type. Changing `render-spells.js` won't break `render-bestiary.js`. |
| Individual `filter-*.js` files (page-level) | Page-specific filter configs. `filter-spells.js` is independent of `filter-items.js`. |
| `data/*.json` (additive changes) | Adding entries to existing arrays is safe. Modifying schema structure is not. |

### High Risk — Do Not Edit Unless Necessary

| File | Why Risky | Lines |
|------|-----------|-------|
| `js/utils.js` | Global utilities used everywhere. Breaks propagate to every page. | 9,551 |
| `js/render.js` | Core rendering engine. All entity display depends on it. | 17,153 |
| `js/parser.js` | Data normalization for all entity types. | 4,508 |
| `js/listpage.js` | Shared list/detail page framework. Every list page inherits from it. | 2,708 |
| `js/navigation.js` | Site-wide navigation. | — |
| `js/filter.js` + `js/filter/` | Core filter framework (not page-specific filters). | — |
| `js/utils-dataloader.js` | Data loading pipeline. Affects all data access. | — |

### Recommended Personal Customization Structure

Create a `_personal/` directory at the repo root:

```
_personal/
├── homebrew/           # Your homebrew JSON content
│   ├── my-spells.json
│   ├── my-items.json
│   └── my-monsters.json
├── scss/               # Personal style overrides
│   └── _personal.scss  # Import this in scss/main.scss
├── js/                 # Personal JS patches (overlay scripts)
│   └── personal-overrides.js
├── notes/              # Dev notes, plans, session logs
└── README.md           # What you changed and why
```

Add `_personal/` to `.gitignore` so it survives rebases cleanly. For homebrew content, you can still use the built-in `homebrew/` directory — but `_personal/homebrew/` gives you a backup that won't conflict.

### Commenting Convention for Personal Modifications

When you must edit core files directly, use this pattern to make changes easy to find and maintain during rebases:

```javascript
// [PERSONAL] <your-initials> <date> — <brief description>
// Original: <original code or "none">
yourModifiedCode();
// [/PERSONAL]
```

Example:
```javascript
// [PERSONAL] DH 2026-03-24 — Add custom homebrew source abbreviation
Parser.SOURCE_JSON_TO_ABR["MyBrew"] = "MB";
// [/PERSONAL]
```

For SCSS:
```scss
// [PERSONAL] DH — Custom dark theme tweaks
.my-custom-class { color: #ff0; }
// [/PERSONAL]
```

This makes `grep -r "\[PERSONAL\]"` instantly locate all your changes.

### Recommended Branching & Rebase Workflow

Your repo is already configured with:
- `origin` → `https://github.com/doover17/scry-5etools.git` (your fork)
- `upstream` → `https://github.com/5etools-mirror-3/5etools-src.git` (source)

**Workflow:**

```bash
# 1. Keep a clean 'main' branch that mirrors upstream exactly
git checkout main
git fetch upstream
git merge upstream/main          # Fast-forward merge (main should always be clean)
git push origin main             # Keep your fork's main up to date

# 2. Do all personal work on a 'personal' branch
git checkout -b personal         # Create once
# ... make your changes ...
git commit -m "personal: description of change"

# 3. When upstream updates, rebase your personal branch
git checkout main
git fetch upstream
git merge upstream/main
git push origin main

git checkout personal
git rebase main                  # Replay your changes on top of latest upstream
# Resolve conflicts if any — your [PERSONAL] comments will help locate your changes
git push origin personal --force-with-lease   # Update your fork's personal branch

# 4. If rebase gets messy, you can always recreate from scratch
git checkout main
git checkout -b personal-v2
# Re-apply your changes manually using grep for [PERSONAL] markers
```

**Branch naming convention:**
- `main` — Always clean mirror of upstream
- `personal` — Your active modifications
- `personal-feature-X` — Experimental feature branches off `personal`

## SECTION 6 — AI-Assisted Development Notes

### Files That Need Summarization Before Editing

These files are too large to hold in working memory. Always read a targeted section before editing:

| File | Size | Strategy |
|------|------|----------|
| `js/render.js` | 17,153 lines | Search for specific method/class names. Don't read the whole file. |
| `js/utils.js` | 9,551 lines | Grep for the specific utility you need. Constants are near the top (`VeCt`). |
| `js/parser.js` | 4,508 lines | Static methods — grep for `Parser.METHOD_NAME`. |
| `js/listpage.js` | 2,708 lines | Search for class names like `SublistManager`, `ListPage`. |
| `data/items.json` | 2.7 MB | Never load entirely. Search for specific item names or properties. |
| `data/deities.json` | 786 KB | Search by name. |
| `data/recipes.json` | 749 KB | Search by name. |

### Chunking Strategy for Large Files

- **`js/render.js`**: Organized by `Renderer` class methods and inner helper classes. Target specific rendering methods (e.g., `_renderEntries`, `_renderTable`). Each method is self-contained.
- **`js/utils.js`**: Grep for specific function names. The `VeCt` constants block is at the top. String prototype extensions are near the top. Page-specific utilities are further down.
- **`js/parser.js`**: Purely static methods on the `Parser` class. Each `Parser.X_TO_Y` mapping or `Parser.parseX()` method is independent. Safe to read/edit one method at a time.
- **Data JSON files**: Use `jq` or targeted grep. Never attempt full-file rewrites. Add entries to arrays with targeted edits.

### Search-and-Patch vs. Full Rewrite

| Approach | When to Use | Examples |
|----------|-------------|---------|
| **Search-and-patch** | Core files, large files, files with high upstream churn | `js/utils.js`, `js/render.js`, `js/parser.js`, large JSON data files |
| **Full rewrite** | Small page-specific files, new files, personal override files | `render-spells.js` (354 lines), personal SCSS files, homebrew JSON |
| **Additive only** | Data files where you're adding content | `homebrew/*.json`, personal content JSON files |

### Conventions an AI Agent Should Follow

1. **ES module syntax:** Use `import`/`export` — the project uses `"type": "module"`.
2. **Class-based patterns:** New page features should extend existing base classes (e.g., `extends SublistManager`, `extends ListPage`).
3. **Static utility classes:** Utility code uses static methods on classes (e.g., `class FilterCommon { static getDamageVulnerableFilter() {...} }`).
4. **Source abbreviations:** Every entity needs a `"source"` field with a known abbreviation. New sources must be registered in `parser.js`.
5. **Naming conventions:**
   - Page JS files match HTML filenames: `spells.html` → `js/spells.js`
   - Render files: `render-{entity}.js`
   - Filter files: `filter-{entity}.js`
   - Data files: `data/{entity}.json` or `data/{entity}/{entity}-{source}.json`
6. **No semicolons after class declarations.** The codebase uses semicolons for statements but not after class/function declarations.
7. **Tab indentation** in JS and HTML (per `.editorconfig`).
8. **Filter factory pattern:** Page-specific filters create `Filter` objects with `header`, `items`, `displayFn`, `displayFnMini`, `displayFnTitle`.
9. **Hash-based routing:** Entity selection uses URL hashes (e.g., `#fireball_phb`). The `VeCt.HASH_*` constants define separators.
10. **`Renderer` is the single rendering pipeline.** All entity display goes through `Renderer` methods. Don't create parallel rendering paths.
